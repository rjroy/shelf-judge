MEETING NOTES: Tournament Ranking Spec Review
Date: 2026-04-06
Attendees: User, Octavia (chronicler)
Project: shelf-judge

DISCUSSION SUMMARY

Octavia presented the tournament ranking specification for review, which adds pairwise ELO-based ranking as a second fitness signal alongside existing axis scores. The spec is comprehensive, covering data model, ELO calculation, normalization, session management, and API surface. The review focused on three open questions that required clarification before approval.

The user and Octavia worked through configuration decisions that make the system more tunable based on real usage. K-factor transition and provisional display thresholds were moved from fixed constants to configurable daemon settings. The normalization reference window (used to map raw ELO to the 1-10 display scale) was similarly made configurable. These changes preserve the ability to ship with sensible defaults while allowing adjustment without code changes.

The spec remains in draft status with all revisions incorporated and ready for implementation planning or design work.

KEY REFINEMENTS

Open Question 1 (K-factor tuning): K-factor threshold moved from 30 to 15 comparisons and made configurable via daemon settings. K-values themselves (32/16) remain fixed constants. Rationale: 30 comparisons across 200 games requires 6,000 forced choices; 15 reduces that to 2,000, making it reachable within realistic usage. Recalculate operation (REQ-TOURN-7) makes future adjustments safe.

Open Question 2 (Normalization reference range): Fixed reference window (1100-1900) made configurable as a half-width value (default 400). Allows users to adjust if ELO scores cluster too narrowly. Rationale: Fixed range is stable and predictable early on; if compression occurs after extensive use, admin can adjust and recalculate without code changes.

Open Question 3 (Session filter UX): Reframed as active design work rather than deferred scope. The spec defines four filter types (name, axis fitness, BGG tag, staleness); the design document will determine how they're surfaced (builder UI, syntax, presets, etc.). Spec requires the capability, design determines the surface.

Configuration updates to requirements:
- REQ-TOURN-6: Threshold 15, configurable via daemon settings
- REQ-TOURN-9: Half-width 400, configurable via daemon settings
- REQ-TOURN-10: Provisional threshold 6, configurable via daemon settings
- REQ-TOURN-12: Staleness default now tracks provisional threshold (6)

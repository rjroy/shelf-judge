MEETING NOTES: Utility Curves Specification Revision (2026-04-07)

SUMMARY

The session focused on revising the Utility Curves specification to address a fundamental concept problem: the previous draft treated normalization as a layer applied on top of the existing fitness formula. The revision reframes curves as *the* normalization mechanism itself. This reshapes how native scales work across all axes. Personal axes have a fixed native scale of 1-10. BGG-derived axes have hardcoded native scales per field (complexity 1-5, community rating 1-10). The curve defines the mapping from native scale to the effective 1-10 rating used in the weighted average fitness calculation.

The conceptual shift eliminated redundancy in the specification. The old REQ-CURVE-23 ("show both axis-scale and BGG-scale") was removed because there is no longer a separate axis scale—the native scale is what the user sees and configures in. BGG complexity no longer uses a hardcoded `weight * 2` normalization; instead, it uses a proper linear stretch from 1-5 to 1-10 by default, with the curve system allowing users to override this with sweet spots, lower-is-better preferences, or asymmetric tolerances. This produces a minor score correction from the old implementation, which was deliberate and documented as REQ-CURVE-6.

Specification changes include: three preference shapes (higher-is-better, lower-is-better, sweet spot with configurable ideal, tolerance, and lean direction), veto thresholds operating on native-scale values, and a complete restructuring of the UI/CLI sections to work in native-scale terms throughout. The spec grew from 25 to 28 requirements. The reasoning for each major decision is preserved in the Context section, particularly regarding why additive aggregation was retained over multiplicative (explicit veto is more legible) and why native scale was promoted to a first-class concept (honest about the transformation happening).

DECISIONS

Curves replace hardcoded BGG mappings. The `resolveBggRating` function will be superseded by the curve system operating on raw native-scale values. Default "higher is better" for all axes preserves personal axis scores while correcting BGG complexity normalization from `weight * 2` (1-5 to 2-10) to linear (1-5 to 1-10).

Three preference shapes instead of four. The brainstorm's Approach 4 proposed linear, plateau, S-curve, and veto. This spec uses linear ("higher is better" and "lower is better"), sweet spot with configurable tolerance and lean, and explicit veto as an axis-level option. The visual curve preview in the UI eliminates the "calibration difficulty" risk identified in the brainstorm by letting users see the effect before committing.

Additive weighted average persists. Multiplicative aggregation was rejected because a single vetoed axis producing 0 is more transparent than a geometric mean silently producing 0.3 with most axes scoring well.

ARTIFACTS

specs/utility-curves.md (revised): 28 requirements organized into nine sections (Native Scale, Preference Shapes, Sweet Spot Configuration, Effective Rating, Veto Thresholds, Fitness Breakdown, Web UI, CLI, BGG-Derived Axes). Includes exit points for future work (custom curve editor, axis templates, redundancy scoring), success criteria separated by verification method, and custom AI validation criteria for curve math, backward compatibility, and native-scale consistency.

OPEN ITEMS

None. Specification approved. Session closed with commit recording the revision.

**Meeting: Review fix-community-stats plan with Octavia**

The session focused on reviewing a seven-step implementation plan to fix empty community stats in the shelf-judge collection profiler. Octavia provided initial assessment of the plan's structure and surfaced three open questions for discussion. The conversation identified a significant flaw in the plan's approach to value normalization and cleaned up speculative infrastructure concerns.

The critical finding: the original plan's `resolveAxisValues` utility was designed to normalize all values to a 1-10 scale for consistency. This was incorrect. Users need to see native values in axis distributions (weight should display as 1-5, not normalized to 1-10). Normalization is a fitness-scoring concern, not a display concern. This required revising Step 1 to return native-scale values, with normalization remaining the responsibility of consumers that need it (fitness service, feature vector encoding). The first open question (profile cache recompute endpoint) was deemed unnecessary complexity for a single-user local app—users can delete the cache file or wait for the next collection change. The second question answered itself once the native-scale decision was made. The third question (outlier detection sensitivity) wasn't a decision point—better data produces better results; it's correct behavior.

The plan was updated: Step 1 redesigned to return native values, Step 5 simplified to "delete ~/.shelf-judge/profile.json," Step 6 tests corrected to expect native-scale results, and all three open questions deleted entirely. The delegation guide and validation step remain unchanged.

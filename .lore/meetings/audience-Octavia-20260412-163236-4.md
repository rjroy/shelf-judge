SHELF JUDGE DESIGN REVIEW: SHELF CAPACITY SPECIFICATION
Meeting with Octavia, 2026-04-12

DISCUSSION SUMMARY

Octavia conducted a comprehensive design review of the shelf capacity specification covering box dimensions, shelf configuration, and overflow calculation. The spec implements three layers: storing physical game dimensions, modeling shelf configuration, and computing capacity overflow. During review, multiple internal contradictions and design gaps surfaced. The specification opens with a core argument that volume totals alone are insufficient because shape-fitting is what matters (a 25-inch vintage box cannot fit in a 14-inch Kallax cube regardless of total volume), but then defines the overflow status using pure volume comparison—the exact limitation the overview identifies as insufficient. Additional structural problems include the `perShelfUtilization` field promising per-shelf capacity information without the game-to-shelf assignment data needed to compute it, the overflow list implicitly treating the entire collection as on all shelves simultaneously (ignoring actual placement), and multiple edge cases that are unverifiable without additional specification (refresh vs. manual dimension override, unconstrained-height shelf communication, exact-capacity boundary conditions).

DECISIONS AND REASONING

User decided not to implement the spec as currently written. The spec either over-complicates a problem that was solved more simply in a prior implementation, or fails to push far enough into the actual solution space. User requested all identified flaws be documented directly in the spec itself rather than addressed through implementation workarounds. User will close this session and reconvene with either new research into the problem domain or return to the original implementation approach. The decision between attempting simpler assignment logic versus reframing the overflow question entirely is deferred to next session.

ARTIFACTS AND DOCUMENTATION

Spec file updated with a "Known Flaws" section documenting seven distinct problems: internal contradiction (volume vs. shape), incomplete field definitions, missing interaction specs, verifiability gaps, inconsistent API/storage representations, untestable success criteria, and undefined sort ordering for BGG version selection. Status line flagged as draft with known issues. One strength identified: the unfittable-games check (six-orientation fit verification) is correctly specified and self-contained and requires no assignment data.

FOLLOW-UP

Session closed pending decision on next approach (research or prior implementation).

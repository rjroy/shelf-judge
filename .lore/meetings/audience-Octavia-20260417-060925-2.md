**Meeting Summary**

The team reorganized the `.lore/` directory structure to address sizing and discoverability issues across plans (23), specs (15), issues (38), and retros (12). The user rejected a previous "cohesion-based" approach as unhelpful and specified that the real problem was lack of a searchable axis within each lifecycle-phase directory. The team identified **feature domain** as the organizing principle, partitioning all directories along product areas: fitness (scoring, prediction, redundancy, utility-curves), collection (filtering, sorting, profiling, search), tournament (bracket mechanics, ranking), bgg (external integration), ui (presentation), infra (daemon, logging, lint), and features (kitchen drawer for wishlist, previously-owned, shelf-capacity, llm-narrative). Cross-cutting documents (mvp.md, mvp specs) remain at the top level. Retros follow a different axis: cleanup/ for periodic maintenance snapshots and incident/ for bug retros.

**Key Decisions**

Feature domain became the organizing axis for all four lifecycle directories. mvp.md and mvp-related specs stay top-level due to their foundational, cross-cutting nature—this decision keeps navigation simple for foundational reference. The features/ subdirectory was explicitly flagged as a "kitchen drawer" pending enough growth to justify domain separation; this acknowledges technical debt without imposing premature structure. Retros use a different axis (cleanup vs incident) because their review purpose differs from the design-to-deployment lifecycle; cleanup captures routine maintenance, incident isolates post-mortems.

**Artifacts Produced**

All 86 files moved across four directories with references validated. Fifty-five files had `related:` fields and inline path references rewritten. One HTML mockup (profile-overview.html) had two inline references fixed. All changes staged as git renames; commit landed with no dangling references.

**Open Items**

The features/ subdirectory is acknowledged as a temporary bucket. Revisit if any contained feature (wishlist, previously-owned, shelf-capacity, llm-narrative) grows enough to justify its own domain at top level.

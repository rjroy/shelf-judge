Rating Interpretation Labels

Enable consistent rating application across all axes by displaying a human-readable interpretation label alongside every numeric rating. Labels anchor the 1-10 scale semantically, helping users calibrate ratings consistently over time and across axes.

Requirements

REQ-RATING-LABELS-1
Define Rating Interpretation Labels
Establish a fixed 1-10 label mapping that applies to all axes uniformly:
Rating Label Semantic Meaning
10 Essential Perfect for this axis. Sets the standard.
9 Definitive Outstanding. No meaningful criticism on this axis.
8 Recommended Clearly good. Worth prioritizing.
7 Very Good Solid. Exceeds baseline but not standout.
6 Good Acceptable. Meets expectations.
5 Fine Neutral. Neither good nor bad.
4 Not Good Noticeable weaknesses.
3 Just Bad Significant problems.
2 Inexplicable Hard to justify except as an outlier.
1 Offensive Actively unacceptable.
REQ-RATING-LABELS-2
Export Label Mapping as a Shared Function
Create a utility function in packages/shared/src/ that maps rating numbers to labels. The function must handle decimal ratings (e.g., from predictions) by rounding to the nearest integer before lookup:
export function getRatingLabel(rating: number | null): string | null { // Returns label string for 1-10 // Rounds decimal ratings (e.g., 5.8 -> 6) to nearest integer // Returns null if rating is null or out of range } export const RATING_LABELS: Record<number, string> = { 1: "Offensive", 2: "Inexplicable", 3: "Just Bad", 4: "Not Good", 5: "Fine", 6: "Good", 7: "Very Good", 8: "Recommended", 9: "Definitive", 10: "Essential" };
Rounding Behavior
Use standard "round half up" (Math.round) so 5.5 becomes 6. This applies to all decimal ratings: BGG-derived (e.g., 5.8 for Complexity), predicted (e.g., 6.3 for prediction), and any future source that produces decimals.
Note
This ensures the mapping is single-source-of-truth and reusable across daemon responses, web components, and CLI output.
REQ-RATING-LABELS-3
Display Label in Score Breakdown Table
Modify the ScoreBreakdown component to show interpretation labels in the rating column. For each breakdown entry with a rating, append the label:
<td class="right"> {entry.rating !== null ? `${entry.rating} ${getRatingLabel(entry.rating)}` : "—"} </td>
Placement
The label appears in the "Effective" column (already displays the numeric rating). Format: 8 Recommended or on two lines if space is tight.
REQ-RATING-LABELS-4
Display Label in Rating Form
Modify the RatingForm component to display the interpretation label as the user sets or changes a rating. The label should update in real-time as the slider/input value changes.
Location
Show the label below or next to the numeric input/slider, updating live as the user adjusts the value.
REQ-RATING-LABELS-5
Display Label for BGG-Derived Ratings
When a rating is auto-populated from BGG (e.g., Community Rating, Complexity), display its interpretation label alongside the numeric value to help the user understand what the BGG rating means before they override it.
Applies to
  • Score breakdown table, for entries where source === "bgg"
  • Rating form, for auto-populated BGG fields
  • Override context display (e.g., "BGG: 5.8 Fine")
REQ-RATING-LABELS-6
No Label for Missing or Null Ratings
When a rating is null (not yet rated), do not display a label. Show only the dash/empty state as currently implemented.

Acceptance Criteria

  • Every numeric rating (1-10) in the app displays its corresponding interpretation label
  • Labels are consistent across all axes and all contexts (breakdown, form, BGG data)
  • The label mapping is defined once in shared code and reused everywhere
  • Rating form shows live label updates as user adjusts values
  • Score breakdown table shows labels for all rated axes
  • BGG-derived ratings show labels to clarify what the BGG value means
  • No labels appear for null/unrated entries
  • All tests pass; no regressions in existing rating or breakdown functionality

Notes

Rationale

The label mapping you provided is calibrated for consistency—it's the mental scale you use when rating. By embedding these labels throughout the UI, we remove the gap between rating intention and rating execution. Instead of "what does a 7 mean again?", users see the label immediately and apply it consistently.

Rounding decimal ratings to integers preserves the user's intent: a predicted 5.8 is "closer to Good (6)" than "closer to Fine (5)", so the label anchors that interpretation visually.

Future Extensions

Axis-specific label overrides (e.g., "Wife will play it" might use different language) are not in scope for MVP but the architecture should allow them. Store labels in the Axis record if customization is added later.

Open Question

Display format in dense tables: should the label appear on the same line as the number (e.g., "8 Recommended") or on a new line? This depends on layout and readability. Current breakdown table is already information-dense; a new line might be clearer. Verify during implementation.
